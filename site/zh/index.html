    01

    从提示到行动

    启动链路本身就是性能架构：轻请求即时返回，重系统按需唤醒，不做无效初始化。

    你按下回车后的第一毫秒

    执行 `claude` 后，进程先做分流判定，不做全量加载。固定顺序是：读 `process.argv`，判定请求级别，再决定是否进入重型启动链路。

    角色出场

    用户看到“一个助手”，系统实际是责任接力链：入口分流、引导初始化、界面承载、回合编排、工具执行。

    把架构看成职责地图，不要看成一团“应用”

    架构理解的关键是建立责任映射。入口层负责分流，bootstrap 负责初始化，REPL 负责交互承载，QueryEngine 负责回合推进，工具层负责外部执行。

    🚪

    🧰

    执行

    🔁

    03

    对话循环

    单次请求会经过一个状态机循环：模型输出、工具调用、结果回写、下一轮决策。

    一个回合，往往包含很多次来回

    示例请求“修掉 auth.ts 的 bug”通常会触发多轮往返：读文件、跑测试、读错误、改文件、复测。用户看到是一回合，系统内部可能已执行多次模型-工具迭代。

    1
    拼上下文

    历史消息、system prompt、用户上下文和工具权限先被装配齐。

    2
    模型给出下一步

    它可能直接回答，也可能发出结构化的 `tool_use` 请求。

    3
    工具执行

    真正的读文件、跑命令、查资源，都在工具层发生。

    4
    结果回写，再继续

    工具结果会被追加回消息流，成为下一轮模型决策的输入。

  toolUseMessages: ToolUseBlock[],
  assistantMessages: AssistantMessage[],
  canUseTool: CanUseToolFn,
  toolUseContext: ToolUseContext,

  let currentContext = toolUseContext
  for (const { isConcurrencySafe, blocks } of partitionToolCalls(...)) {

    } else {

    这就是为什么“同时读 5 个文件”通常很快，而“连续改 5 个文件”不会被强行并发。

    04

    工具与外部世界

    模型只负责决策，工具才负责执行。文件系统、shell、网页和远程服务访问都发生在工具/连接器边界。

    先把工具分门别类

    📁

    文件系统工具

    Read、Edit、Write、Glob、Grep，负责本地代码库上的观察与修改。

    🧪

    执行工具

    Bash、PowerShell 等，负责把“想法”落成可执行动作。

    🌐

    Web 工具

    Fetch、Search 等，用来获取本地没有的外部信息。

    🧩

    MCP 连接器

    通过协议接入远程能力，让数据库、浏览器、IDE 等都能以工具形式暴露给代理。

    一次工具调用到底怎么走

    1
    模型发出 `tool_use`

    它不会自己执行动作，只会提出“我想调用某个工具”的结构化请求。

    2
    QueryEngine 检查权限

    调用先过权限门，再决定是允许、询问还是拒绝。

    3
    工具或连接器执行

    真正的文件读取、命令执行、远程请求在这一层发生。

    4
    结果写回回合

    执行结果会变成 `tool_result` 再回到下一轮模型上下文里。

    ⚠️
    “模型坏了”通常是错误诊断

    如果文件没读到、命令没跑通、远程会话过期，根因通常在工具层或连接器层，优先检查执行边界。

    MCP 会话过期是一个真实而常见的失败模式

    `source/claude-code-source/src/services/mcp/client.ts`（第 193-205 行）

    CODE 
    export function isMcpSessionExpiredError(error: Error): boolean {
  const httpStatus = 'code' in error ? (error as Error & { code?: number }).code : undefined
  if (httpStatus !== 404) return false
  return (
    error.message.includes('"code":-32001') ||
    error.message.includes('"code": -32001')
  )
}
    中文解读

    一个 404 本身是不够的，它可能是 URL 错，也可能是 session 真过期了。

    所以运行时还要进一步检查错误消息里是否包含 JSON-RPC 的 `-32001`。

    这一步很关键，因为只有把失败类型判准，后续才知道该走“重连重试”还是“彻底报错”。

    连接器元数据本身也会吞上下文预算

    MCP 风险不只在会话状态，也在描述体积。OpenAPI 生成连接器常把大量接口文档塞进工具描述；不做截断时，元数据本身就会侵占上下文预算。

    MAX_MCP_DESCRIPTION_LENGTH 把工具或服务描述限制在 2048 个字符，避免文档洪水挤占上下文
    MCP_TOOL_TIMEOUT 远程工具调用的超时窗口可配置，避免无限等待
    McpAuthError 把鉴权失败显式建模，便于上层走恢复路径
    📦
    连接器同样承担上下文预算管理职责

    工具越多、描述越胖、超时时间越乱，代理越容易在非模型层面失控。

    05

    权限与信任

    高能力需要清晰边界。权限系统的价值在于把执行权切到正确粒度，并保持可控。

    权限模式，本质上是不同信任档位

    default 敏感动作默认先询问
    acceptEdits 偏向自动接受编辑类动作
    bypassPermissions 广义自动放行，风险最高
    plan 以规划为主，执行能力更收敛

    安全策略取决于粒度是否正确。规则越具体，授权面越可控。

    规则匹配需要足够细，不能停留在“允许 Bash”这种粗粒度表达

    Bash(git status) 只批准一个具体命令，粒度最细
    Bash(git:*) 批准一类 git 命令，边界变宽
    Bash(*) 几乎等于把整个 shell 提前放行，风险骤增
    ⚠️
    权限规则最怕过宽

    一条看起来“很省事”的广泛授权，通常意味着你把本来该逐次审查的能力直接整体放出去了。

    什么样的 Bash 规则会被判成危险

    速度技巧与排障

    速度来自系统性编排，不来自单点技巧：可并行就并行，可延后就延后，可观测就留证据。

    并行启动会直接减少等待时间

    startup profiler 提供阶段证据

    init_time 看初始化逻辑是否拖慢启动
    settings_time 看设置加载是否成为瓶颈

    🧭
    诊断性能，先问“慢在哪一段”

    启动慢、回合慢、上下文管理慢，这三种慢的修法根本不同。先定位阶段，再谈优化。

    上下文工程

    Claude Code 的关键不在“写一句更聪明的 prompt”，而在于它如何在模型开口前，把系统提示、工具、规则文件、记忆和历史上下文组装成一整套运行时上下文。

    prompt 只是露在外面的尖端

    在真实运行里，模型会看到拼装后的上下文栈：system prompt、工具 schema、`CLAUDE.md` 层级、memory 文件，以及经过压缩后的会话历史。所谓 上下文工程，本质上就是这套拼装系统。

    📘

    System prompt

    定义角色、约束、安全边界和工具可用性。

    🗂️

    规则文件

    `CLAUDE.md`、`.claude/CLAUDE.md`、rules 文件共同决定 repo 级行为。

    🧠

    记忆与历史

    memory、session history 和 compact 结果决定下一轮“还能记住什么”。

    💡
    很多“prompt 不行”的问题，其实是上下文栈组装错了

    如果代理忽略 repo 规范、忘了前面结论、或者行为突然变形，第一怀疑对象通常是上下文装配链。

    成本与 Token 经济学

    Claude Code 把成本、Token、上下文压力拆成独立账本。看懂这些账本的计算位置，才能做稳定的预算和运行策略。

    同一轮响应可以同时影响美元成本、Token 分类计数、时延指标、代码改动规模。成本治理是系统问题，单看一个数字会丢失关键信号。

    📏

    运行账本

    Memory 设计：持久化、约束与同步

    Claude Code 的 memory 体系由索引文件、类型化记忆文件、路径安全校验和团队同步组成。目标是稳定召回、可控规模、可验证边界。

    Agent 团队与通信系统

    多代理协作是否稳定，取决于协调协议、消息传输、状态队列和通知回流是否一致。Claude Code 把这些能力都做成了显式运行时结构。

    🎯

    主动模式、后台会话与定时代理

    Claude Code 的自治执行有三条主路径：proactive tick、后台主会话任务、cron 定时任务。它们共用消息队列，并用 agent 作用域做精确路由。

    先建立执行模型：三种自治回路

    ⏱️

    Proactive Tick 回路

    在队列空闲时注入隐藏 `<tick>` prompt，让模型持续推进任务。

    🧵

    后台主会话任务

    把当前主线程查询转成后台任务，前台继续可交互，输出写入独立 sidechain transcript。

    🗓️

    Cron 定时任务

    从 `.claude/scheduled_tasks.json` 触发 prompt，进入 `later` 优先级队列，按 agentId 分发。

    工程关注点
