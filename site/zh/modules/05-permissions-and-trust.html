权限与信任

高能力需要清晰边界。权限系统的价值在于把执行权切到正确粒度，并保持可控。

权限模式，本质上是不同信任档位

default 敏感动作默认先询问
acceptEdits 偏向自动接受编辑类动作
bypassPermissions 广义自动放行，风险最高
plan 以规划为主，执行能力更收敛

安全策略取决于粒度是否正确。规则越具体，授权面越可控。

规则匹配需要足够细，不能停留在“允许 Bash”这种粗粒度表达

Bash(git status) 只批准一个具体命令，粒度最细
Bash(git:*) 批准一类 git 命令，边界变宽
Bash(*) 几乎等于把整个 shell 提前放行，风险骤增
⚠️
权限规则最怕过宽

一条看起来“很省事”的广泛授权，通常意味着你把本来该逐次审查的能力直接整体放出去了。

什么样的 Bash 规则会被判成危险

`source/claude-code-source/src/utils/permissions/permissionSetup.ts`（第 94-147 行）

CODE 
export function isDangerousBashPermission(toolName: string, ruleContent: string | undefined): boolean {
  if (toolName !== BASH_TOOL_NAME) return false
  if (ruleContent === undefined || ruleContent === '') return true
  const content = ruleContent.trim().toLowerCase()
  if (content === '*') return true
  for (const pattern of DANGEROUS_BASH_PATTERNS) { ... }
  return false
}
中文解读

如果 Bash 规则连内容都没有，本质上就是“整类动作全放行”。

单独一个 `*` 也一样，它代表的是近乎无限扩张的匹配范围。

后面还会继续用危险模式表去兜住解释器、shell 和更宽的执行入口。

危险模式表和 auto mode 断路器

`source/claude-code-source/src/utils/permissions/dangerousPatterns.ts`（第 18-42、44-52 行）

CODE 
export const CROSS_PLATFORM_CODE_EXEC = [
  'python', 'node', 'deno', 'tsx', 'ruby', 'perl', 'php', 'lua',
  'npx', 'bunx', 'npm run', 'yarn run', 'pnpm run', 'bun run',
  'bash', 'sh', 'ssh',
] as const

export const DANGEROUS_BASH_PATTERNS: readonly string[] = [
  ...CROSS_PLATFORM_CODE_EXEC, 'zsh', 'fish', 'eval', 'exec', 'env', 'xargs', 'sudo',
]
中文解读

这份列表的共同点很简单：它们都能把“允许一个命令”扩张成“允许执行任意代码”。

解释器、shell、包运行器、提权命令都在里面，因为这些入口一旦被宽泛放行，auto mode 的分类器几乎形同虚设。

`source/claude-code-source/src/utils/permissions/autoModeState.ts`（第 27-33 行）

CODE 
export function setAutoModeCircuitBroken(broken: boolean): void {
  autoModeCircuitBroken = broken
}

export function isAutoModeCircuitBroken(): boolean {
  return autoModeCircuitBroken
}
中文解读

auto mode 存在可触发的安全断路器。断路之后，上层逻辑会阻止它悄悄重新开启。

这是一种非常典型的“防止错误配置自动复活”的防线。
