主动模式、后台会话与定时代理

Claude Code 的自治执行有三条主路径：proactive tick、后台主会话任务、cron 定时任务。它们共用消息队列，并用 agent 作用域做精确路由。

先建立执行模型：三种自治回路

⏱️

Proactive Tick 回路

在队列空闲时注入隐藏 `<tick>` prompt，让模型持续推进任务。

🧵

后台主会话任务

把当前主线程查询转成后台任务，前台继续可交互，输出写入独立 sidechain transcript。

🗓️

Cron 定时任务

从 `.claude/scheduled_tasks.json` 触发 prompt，进入 `later` 优先级队列，按 agentId 分发。

🧭
工程关注点

核心是队列可见性、任务归属、执行隔离。自治能力上线前，先验证这三项。

中文解读

定时任务有两条通路：带 `agentId` 的任务推给对应 teammate；无 `agentId` 的任务进入主线程队列。

子代理只能消费发给自己的 `task-notification`，用户 prompt 固定走主线程。

`isMeta: true` 避免系统自动触发消息污染常规会话视图。

中文解读

本地定时任务依赖本地文件和进程内 scheduler，事件类型包含 `fire` 与 `missed`。

远程定时代理通过 `/schedule` 技能驱动云端 trigger API，受远程会话策略控制。

排障时先判断任务属于本地链路还是远程链路，再选日志和配置入口。

1
先看触发来源

`.claude/scheduled_tasks.json` 是本地链路，`/schedule` 是远程链路。

2
再看运行环境

本地进程队列执行，或云端隔离环境执行。

3
最后看权限边界

本地看队列/任务状态，远程看账号、环境、连接器和策略。
