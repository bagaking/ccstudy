上下文工程

Claude Code 的关键不在“写一句更聪明的 prompt”，而在于它如何在模型开口前，把系统提示、工具、规则文件、记忆和历史上下文组装成一整套运行时上下文。

prompt 只是露在外面的尖端

在真实运行里，模型从来不是只看用户一句话。它看到的是拼装后的上下文栈：system prompt、工具 schema、`CLAUDE.md` 层级、memory 文件、以及经过压缩后的会话历史。所谓 上下文工程，本质上就是这套拼装系统。

📘

System prompt

定义角色、约束、安全边界和工具可用性。

🗂️

规则文件

`CLAUDE.md`、`.claude/CLAUDE.md`、rules 文件共同决定 repo 级行为。

🧠

记忆与历史

memory、session history 和 compact 结果决定下一轮“还能记住什么”。

💡
很多“prompt 不行”的问题，其实是上下文栈组装错了

如果代理忽略 repo 规范、忘了前面结论、或者行为突然变形，第一怀疑对象通常不是模型，而是上下文装配链。

CLAUDE.md 本身就是分层系统

`source/claude-code-source/src/utils/claudemd.ts`（第 1-16 行）

CODE 
/**
 * Files are loaded in the following order:
 *
 * 1. Managed memory (eg. /etc/claude-code/CLAUDE.md) - Global instructions for all users
 * 2. User memory (~/.claude/CLAUDE.md) - Private global instructions for all projects
 * 3. Project memory (CLAUDE.md, .claude/CLAUDE.md, and .claude/rules/*.md in project roots) - Instructions checked into the codebase
 * 4. Local memory (CLAUDE.local.md in project roots) - Private project-specific instructions
 *
 * Files are loaded in reverse order of priority, i.e. the latest files are highest priority
 * with the model paying more attention to them.
 */
中文解读

`CLAUDE.md` 不是一个文件，而是一组不同层级的指令源。

系统级、用户级、项目级、本地级，一层比一层更贴近当前工作上下文。

这也是为什么“为什么在这个 repo 里行为不一样”通常要先问：到底哪一层规则在生效。

中文解读

这里既要保证找到正确的规则文件，也要避免每一轮都重新全量扫描。

所以 `getMemoryFiles` 本身被 memoize 了，说明上下文工程同样要考虑性能。

对 agent 来说，“什么该被看到”与“多快被组装出来”是同一问题的两面。
