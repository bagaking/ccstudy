Memory 设计：持久化、约束与同步

Claude Code 的 memory 体系由索引文件、类型化记忆文件、路径安全校验和团队同步组成。目标是稳定召回、可控规模、可验证边界。

整体结构：索引、类型、召回、同步

🗂️

索引入口

MEMORY.md 承担目录索引角色，始终加载到上下文，并受尺寸上限约束。

🏷️

类型化文件

每条记忆单独成文件，通过 frontmatter 标记类型：user、feedback、project、reference。

🔁

团队同步

团队记忆按仓库范围同步，采用 pull-first + 校验和增量 push。

📌
设计原则

memory 只保存无法从当前代码和 git 历史直接推导的信息，避免重复存档与上下文噪音。

MEMORY.md 被定义为紧凑索引，容量有硬上限。

团队记忆写入采用两阶段路径约束

`source/claude-code-source/src/memdir/teamMemPaths.ts`

CODE 
export async function validateTeamMemKey(relativeKey: string): Promise<string> {
  sanitizePathKey(relativeKey)
  const fullPath = join(getTeamMemPath(), relativeKey)
  const resolvedPath = resolve(fullPath)
  if (!resolvedPath.startsWith(getTeamMemPath())) throw new PathTraversalError(...)
  const realPath = await realpathDeepestExisting(resolvedPath)
  if (!(await isRealPathWithinTeamDir(realPath))) throw new PathTraversalError(...)
  return resolvedPath
}
中文解读

第一阶段做字符串级拦截：空字节、编码穿越、反斜杠、绝对路径、`..` 逃逸。

第二阶段做真实路径校验：解析最深存在祖先的 realpath，再验证仍位于 team 目录内。

该组合策略覆盖符号链接逃逸和悬挂链接写入风险。

🛡️
仅靠 resolve 不够

resolve 可以处理 `..`，但无法识别符号链接目标是否越界。realpath 包含校验必须保留。

召回阶段先扫描头部信息，再用 side query 选择最多 5 条高相关记忆。

同步阶段先 pull，再监听文件变化，2 秒去抖后执行 push。

3
增量提交
