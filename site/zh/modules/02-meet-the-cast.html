02

角色出场

用户看到“一个助手”，系统实际是责任接力链：入口分流、引导初始化、界面承载、回合编排、工具执行。

把架构看成职责地图，不要看成一团“应用”

架构理解的关键是建立责任映射。入口层负责分流，bootstrap 负责初始化，REPL 负责交互承载，QueryEngine 负责回合推进，工具层负责外部执行。

分流

🚪
cli.tsx 读取 argv，判定请求类型，选择启动分支。

引导

🧰
main.tsx 组装权限、工具、命令、状态和启动并发链路。

交互

🖥️
App + REPL 承载终端壳层与实际 prompt/response 交互面。

执行

🔁
QueryEngine 推进回合循环、调度工具，并把结果重新喂回模型。
点击角色，不只是看职责摘要，也能直接打开对应源码。

显示层也是按需加载的

`source/claude-code-source/src/replLauncher.tsx`（第 12-21 行）

CODE 
export async function launchRepl(...): Promise<void> {
  const { App } = await import('./components/App.js');
  const { REPL } = await import('./screens/REPL.js');
  await renderAndRun(root, <App {...appProps}><REPL {...replProps} /></App>);
}
中文解读

交互式路径真正需要时，才去加载 App 壳层和 REPL 视图。

REPL 被包在 App 里面，因此它复用了会话级上下文，并以统一壳层运行。

这就是典型的壳层与屏幕分离：外层负责全局，内层负责当前交互。

工具目录本身就在讲“专人专岗”

source/claude-code-source/src/tools/ 每个工具只负责一种能力
BashTool/ 跑 shell 命令
FileReadTool/ 读取本地文件
FileEditTool/ 以补丁方式改文件
MCPTool/ 调用远程 MCP 服务
WebFetchTool/ 抓网页内容
💡
“关注点分离”是直接提升协作精度的操作杠杆

一旦职责边界清楚，你以后给 AI 的需求就能精确很多，比如“改 QueryEngine 的回合策略，不要碰 REPL 渲染”。
