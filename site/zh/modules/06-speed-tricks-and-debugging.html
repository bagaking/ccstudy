速度技巧与排障

速度来自系统性编排，不来自单点技巧：可并行就并行，可延后就延后，可观测就留证据。

并行启动不是锦上添花，而是直接省等待

`source/claude-code-source/src/main.tsx`（约 1919-1932 行）

CODE 
if (process.env.CLAUDE_CODE_ENTRYPOINT !== 'local-agent') {
  initBuiltinPlugins();
  initBundledSkills();
}
const setupPromise = setup(preSetupCwd, permissionMode, ...);
const commandsPromise = worktreeEnabled ? null : getCommands(preSetupCwd);
const agentDefsPromise = worktreeEnabled ? null : getAgentDefinitionsWithOverrides(preSetupCwd);
commandsPromise?.catch(() => {});
agentDefsPromise?.catch(() => {});
await setupPromise;
中文解读

先把极轻的内存内初始化做掉，然后立刻并行启动 `setup`、命令加载和 agent 定义加载。

这里的关键不是某一行代码，而是顺序设计：先完成必要前置，再让独立重任务重叠执行。

这类优化没有“魔法感”，但对启动体感极其有效。

startup profiler 给你的是阶段证据，不是情绪

import_time 看早期 import 与模块装载是否膨胀
init_time 看初始化逻辑是否拖慢启动
settings_time 看设置加载是否成为瓶颈
total_time 看整条链路总耗时
🧭
诊断性能，先问“慢在哪一段”

启动慢、回合慢、上下文管理慢，这三种慢的修法根本不同。先定位阶段，再谈优化。

中文解读

每次代理改文件时，系统都可以留下与 message 对应的快照。

这不是“锦上添花”的日志，而是后续回滚和排障的硬证据。

最多保留 100 个快照，够用，而且不会无限膨胀。

auto-compact：上下文靠近上限时的缓冲策略

`source/claude-code-source/src/services/compact/autoCompact.ts`（第 62-76 行）

CODE 
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000

export function getAutoCompactThreshold(model: string): number {
  const effectiveContextWindow = getEffectiveContextWindowSize(model)
  const autocompactThreshold =
    effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS
  return autocompactThreshold
}
中文解读

系统不会等上下文完全撞线才处理，而是提前留出缓冲带。

当历史逼近阈值时，旧内容会被压缩总结，为后续回合腾出空间。

这也是长会话能继续跑下去的关键基础设施之一。
