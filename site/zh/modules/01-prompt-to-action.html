01

从提示到行动

启动链路本身就是性能架构：轻请求即时返回，重系统按需唤醒，不做无效初始化。

你按下回车后的第一毫秒

执行 `claude` 后，进程先做分流判定，不做全量加载。固定顺序是：读 `process.argv`，判定请求级别，再决定是否进入重型启动链路。

1
读参数

启动先确认用户意图。只有命中重路径时，才加载整套模块。

2
尽早分流

`--version` 这类轻请求直接走快路径并立即返回。

3
复杂路径才打点

真正要进重型启动流程时，才开始记录性能检查点。

快路径：版本号几乎零成本返回

`source/claude-code-source/src/entrypoints/cli.tsx`（第 34-48 行）

CODE 
const args = process.argv.slice(2);

// Fast-path for --version/-v: zero module loading needed
if (args.length === 1 && (args[0] === '--version' || args[0] === '-v' || args[0] === '-V')) {
  // MACRO.VERSION is inlined at build time
  console.log(`${MACRO.VERSION} (Claude Code)`);
  return;
}

const {
  profileCheckpoint
} = await import('../utils/startupProfiler.js');
profileCheckpoint('cli_entry');
中文解读

先拿到原始命令行参数。

如果用户只问版本号，那这就是轻请求，不值得唤醒重系统。

直接打印构建时注入的版本并返回。

只有真正的会话路径，才去动态导入 startup profiler 并开始打点。

💡
`MACRO.VERSION` 在编译期注入

这意味着版本号读取几乎没有额外 runtime 成本。值在构建时就已经烘焙进产物里了。

startup profiler 把“感觉慢”拆成具名阶段

`source/claude-code-source/src/utils/startupProfiler.ts`（第 50-54 行）

CODE 
import_time: ['cli_entry', 'main_tsx_imports_loaded'],
init_time: ['init_function_start', 'init_function_end'],
settings_time: ['eagerLoadSettings_start', 'eagerLoadSettings_end'],
total_time: ['cli_entry', 'main_after_run'],
中文解读

这里把启动切成具名区间，直接定位具体耗时阶段。

每个阶段都由两个 checkpoint 定义，因此可以非常明确地追责到某一段耗时。

这套方式能直接服务性能诊断，结论来自阶段证据。

🧪
这套诊断工具可以直接拿来用

你现在就可以跑 `CLAUDE_CODE_PROFILE_STARTUP=1 claude --version`，看启动到底慢在 import、init 还是 settings。

分阶段加载链路

真正的重路径是分层推进的：`cli.tsx` 先分流，profiler 负责观测，`main.tsx` 负责重型 bootstrap，随后 `setup()`、`getCommands()`、`getAgentDefinitions()` 再并行展开。
