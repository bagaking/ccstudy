Hooks：运行时策略引擎

Hooks 是运行时策略层：接收生命周期事件、并行执行、合并决策，并在工具调用、停止边界与队友任务状态上执行准入与阻断。

先建立分层模型：事件契约、匹配器、执行器、策略输出

🧾

类型化事件契约

`HookInputSchema` 明确每个事件载荷，包括 `Stop`、`SubagentStop`、`TeammateIdle`、`TaskCompleted`、`PermissionRequest`。

🎯

匹配与来源合并

`getMatchingHooks()` 合并 snapshot、注册回调、plugin/skill、session hooks，再执行 matcher 与 `if` 过滤。

⚙️

并行执行聚合

`executeHooks()` 并行运行 command/prompt/agent/http/function/callback，多源结果汇总为单一决策流。

🚦

策略作用面

可输出 `permissionBehavior`、`updatedInput`、附加上下文，也可通过 exit code 2 触发阻断或停止继续执行。

🧠
工程定位

Hooks 位于控制平面，直接影响执行路径和权限决策。日志只是其中一部分副产物。

生命周期/事件模型来自 SDK schema，边界清晰可审计

事件集合是固定枚举，新增能力必须显式进入 schema。

队友空闲和任务完成属于独立事件，不依赖隐式上下文拼接。

执行语义：并行扇出 + 统一结果归类（阻断与非阻断）

中文解读

并行执行降低整体等待时间，结果由聚合层统一处理。

exit code `2` 代表策略阻断；其他非零返回码记录为非阻断错误。

异步 hook 可后台继续执行，前台决策面保持可控延迟。

blocking exit code 2 或显式 block 决策，产出 `hook_blocking_error`
non_blocking_error 执行/校验失败，保留可观测性，不直接触发拒绝
preventContinuation 停止当前继续链路并带 stop reason，可独立于 blocking 使用

中文解读

交互模式下未建立 trust 时，hooks 不执行。

hook 返回 `allow` 后仍要走规则复核，deny/ask 规则仍可生效。

SDK 场景下 PermissionRequest hooks 与宿主权限弹窗并发竞速，先完成者生效。

1
先过执行门禁

managed-only、disable-all、trust 三层先决定 hooks 是否可运行。

2
再做决策合并

并行结果按优先级合并（`deny > ask > allow`），并可改写输入。

3
最后权限落地

工具执行前仍受规则系统约束，hook 不能绕过核心权限边界。

中文解读

停止阶段是串联策略链，前一阶段可提前终止后续流程。

队友进入 idle 前，会先校验其在途任务是否允许完成。

任务完成阻断会直接影响任务状态写入结果，属于硬约束。

🛑
排障优先级

出现“任务无法完成”时，先查 `TaskCompleted` hook 输出，再看 Stop/TeammateIdle 链路，避免误判为任务系统写入异常。
