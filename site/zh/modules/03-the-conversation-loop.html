对话循环

单次请求会经过一个状态机循环：模型输出、工具调用、结果回写、下一轮决策。

一个回合，往往包含很多次来回

示例请求“修掉 auth.ts 的 bug”通常会触发多轮往返：读文件、跑测试、读错误、改文件、复测。用户看到是一回合，系统内部可能已执行多次模型-工具迭代。

1
拼上下文

历史消息、system prompt、用户上下文和工具权限先被装配齐。

2
模型给出下一步

它可能直接回答，也可能发出结构化的 `tool_use` 请求。

3
工具执行

真正的读文件、跑命令、查资源，都在工具层发生。

4
结果回写，再继续

工具结果会被追加回消息流，成为下一轮模型决策的输入。

这里进入的是一段异步消息流，回合在消息边界上持续推进。

回合推进需要完整上下文：历史消息、规则、工具权限、预算上限。

`for await` 说明系统在逐条消费模型和工具回来的事件，最终输出来自整段流式推进。

并发读，串行写

CODE 
export async function* runTools(
  toolUseMessages: ToolUseBlock[],
  assistantMessages: AssistantMessage[],
  canUseTool: CanUseToolFn,
  toolUseContext: ToolUseContext,
): AsyncGenerator<MessageUpdate, void> {
  let currentContext = toolUseContext
  for (const { isConcurrencySafe, blocks } of partitionToolCalls(...)) {
    if (isConcurrencySafe) {
      // Run read-only batch concurrently
      for await (const update of runToolsConcurrently(...)) {
        yield { message: update.message, newContext: currentContext }
      }
    } else {
      // Run non-read-only batch serially
      for await (const update of runToolsSerially(...)) {
        yield { message: update.message, newContext: currentContext }
      }
    }
  }
}
中文解读

工具不会一股脑乱跑，系统会先按风险和语义把调用分批。

只读工具可以并发，因为顺序通常不重要；改文件、跑命令这类动作则必须串行。

这就是为什么“同时读 5 个文件”通常很快，而“连续改 5 个文件”不会被强行并发。

上下文快满时，系统会主动压缩

`source/claude-code-source/src/services/compact/autoCompact.ts`（第 62-76 行）

CODE 
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000

export function getAutoCompactThreshold(model: string): number {
  const effectiveContextWindow = getEffectiveContextWindowSize(model)
  const autocompactThreshold =
    effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS
  return autocompactThreshold
}
中文解读

长会话不会无限膨胀到把上下文窗口顶爆。

系统会在逼近阈值时主动 compact，把旧历史压缩成摘要，给后续回合腾空间。

所以“模型像是还记得前文”本质上往往是“它重新读了保留下来的压缩历史”。
