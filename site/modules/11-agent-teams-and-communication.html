Agent Teams and Communication

Multi-agent behavior works only when coordination policy, transport, UI queues, and message formats align. Claude Code implements all four as explicit runtime surfaces.

Communication is architecture, not a side channel

🎯

Coordinator policy

The coordinator prompt defines who delegates, who verifies, and how worker outputs are interpreted.

📬

Mailbox transport

Teammates exchange protocol messages through inbox files with lock-based serialization.

🧭

Runtime queues

REPL and AppState keep dedicated queues for worker permission handshakes and sandbox approvals.

📡

Headless parity

`print.ts` drains team notifications and feeds them back into the model loop, matching interactive behavior.

💡
A worker team is a protocol system

Without structured routing, typed messages, and queue ownership, multi-agent runs degrade into duplicated work and stale decisions.

Each agent has a deterministic inbox path under `.claude/teams/{team}/inboxes/`.

Concurrent writers are serialized with lock files and retry backoff, so messages do not clobber each other.

Structured protocol message types are filtered so they route to handlers instead of leaking into raw model context.

⚠️
Transport integrity is part of model quality

If mailbox writes race or protocol messages are misrouted, model reasoning degrades even when prompts are correct.

Workers do not approve their own restricted network actions; they forward requests to the leader.

The leader receives queued requests in UI, responds, and sends decisions back over mailbox transport.

Both sides track pending state, so users can see whether work is blocked on communication.
