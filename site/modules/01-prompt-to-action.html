From Prompt to Action

Startup is staged on purpose. Cheap requests pay almost zero cost, and heavy systems only wake up when needed.

The Millisecond After You Press Enter

When you run `claude`, the process starts by reading `process.argv`. It does not load the whole agent first. It asks one routing question: is this a tiny fast-path request or a full interactive session?

1
Read args

Parse command-line input and classify intent.

2
Branch early

Short commands return immediately with no heavy imports.

3
Profile real startup

Only non-trivial flows turn on startup checkpoints.

Fast Path: Zero Heavy Imports for Version

`source/claude-code-source/src/entrypoints/cli.tsx` (lines 34-48)

CODE 
const args = process.argv.slice(2);

// Fast-path for --version/-v: zero module loading needed
if (args.length === 1 && (args[0] === '--version' || args[0] === '-v' || args[0] === '-V')) {
  // MACRO.VERSION is inlined at build time
  console.log(`${MACRO.VERSION} (Claude Code)`);
  return;
}

const {
  profileCheckpoint
} = await import('../utils/startupProfiler.js');
profileCheckpoint('cli_entry');
PLAIN ENGLISH

Grab command-line args first.

If request is only version, print and exit immediately.

No heavy startup modules are loaded for that path.

Only real sessions import the startup profiler.

💡
Compile-time constant, runtime savings

`MACRO.VERSION` is injected at build time, so version reads are effectively free at runtime.

Startup is measured in named windows, not vague “slow” feelings.

Each phase is a pair of checkpoints.

That gives you blame precision when startup regresses.
