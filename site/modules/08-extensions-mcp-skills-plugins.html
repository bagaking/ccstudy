Extensions: MCP, Skills, Plugins

Claude Code does not grow through one extension surface. It has three: protocol-level connectors, prompt-shaped skills, and installable plugins with lifecycle management.

Three extension surfaces, three different jobs

🌐

MCP

Use this when you need the agent to talk to an external capability through a protocol: browser, database, remote system, IDE bridge.

🧠

Skills

Use this when you want a reusable workflow or instruction bundle that shapes how the agent approaches a class of tasks.

📦

Plugins

Use this when you want packaged distribution, install/update/uninstall flows, marketplace surfacing, and explicit lifecycle management.

💡
These are not interchangeable abstractions

A workflow prompt, a remote protocol connector, and an installable package solve different problems. Most bad extension designs start by choosing the wrong surface.

Bundled skills and plugins are registered before command loading

`source/claude-code-source/src/main.tsx` (lines 1919-1928)

CODE 
// Register bundled skills/plugins before kicking getCommands() — they're
// pure in-memory array pushes (<1ms, zero I/O) that getBundledSkills()
// reads synchronously. Previously ran inside setup() after ~20ms of
// await points, so the parallel getCommands() memoized an empty list.
if (process.env.CLAUDE_CODE_ENTRYPOINT !== 'local-agent') {
  initBuiltinPlugins();
  initBundledSkills();
}
const setupPromise = setup(preSetupCwd, permissionMode, allowDangerouslySkipPermissions, worktreeEnabled, worktreeName, tmuxEnabled, sessionId ? validateUuid(sessionId) : undefined, worktreePRNumber, messagingSocketPath);
const commandsPromise = worktreeEnabled ? null : getCommands(preSetupCwd);
PLAIN ENGLISH

Extensions are part of startup topology, not a late afterthought.

Bundled skills and built-in plugins are registered before command loading so the command layer sees the right inventory immediately.

In other words: extension surfaces are first-class citizens in the boot sequence.

Skills are aggregated from multiple sources

`source/claude-code-source/src/commands.ts` (lines 359-385)

CODE 
const [skillDirCommands, pluginSkills] = await Promise.all([
  getSkillDirCommands(cwd).catch(err => {
    logError(toError(err))
    logForDebugging('Skill directory commands failed to load, continuing without them')
    return []
  }),
  getPluginSkills().catch(err => {
    logError(toError(err))
    logForDebugging('Plugin skills failed to load, continuing without them')
    return []
  }),
])
const bundledSkills = getBundledSkills()
const builtinPluginSkills = getBuiltinPluginSkillCommands()
PLAIN ENGLISH

“Skills” do not come from one folder. The runtime merges skill-directory commands, plugin-provided skills, bundled skills, and built-in plugin skills.

It also treats failure defensively: one broken skill source should not take down the entire command surface.

This is extension-system design in practice: layered discovery plus graceful degradation.

Plugins are a full product surface, not a hidden hook

`source/claude-code-source/src/main.tsx` (lines 4148-4172)

CODE 
const pluginCmd = program.command('plugin').alias('plugins').description('Manage Claude Code plugins').configureHelp(createSortedHelpConfig());
pluginCmd.command('validate <path>').description('Validate a plugin or marketplace manifest')
pluginCmd.command('list').description('List installed plugins').option('--json', 'Output as JSON')
const marketplaceCmd = pluginCmd.command('marketplace').description('Manage Claude Code marketplaces').configureHelp(createSortedHelpConfig());
marketplaceCmd.command('add <source>').description('Add a marketplace from a URL, path, or GitHub repo')
PLAIN ENGLISH

Plugins are not just loadable code blobs. They come with explicit CLI management, validation, listing, marketplace handling, and install/update flows.

That means the extension surface is treated like a product with lifecycle rules, not just a hacky escape hatch.

MCP Protocol-based capability hookup for external systems
Skills Reusable task-shaping instructions and workflows
Plugins Packaged, managed extension units with installation lifecycle
