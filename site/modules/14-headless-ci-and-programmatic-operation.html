Headless CI and Programmatic Operation

Headless mode is a protocol surface, not a UI shortcut. `--print`, stream-json I/O, structured output tooling, and session resume semantics are designed to keep automated runs deterministic and recoverable.

Execution map for automation: one runtime, four contracts

🖨️

Headless Entry

`-p/--print` switches to non-interactive execution and routes into `runHeadless(...)`.

📡

NDJSON Protocol

`--output-format=stream-json` uses framed stdout messages plus control request/response flow.

🧱

Schema Output

`--json-schema` injects `StructuredOutput` tool validation and retry limits.

🔁

State Continuity

`--continue`/`--resume` reuse transcript state with explicit forking and cursor controls.

🧭
Operational rule

In CI, prefer machine-parseable outputs and explicit stop conditions (`maxTurns`, budget, schema). Do not rely on terminal-only affordances.

Headless mode is explicit: `--print` is the switch that routes startup into the non-interactive runtime.

Stream input/output must be paired as `stream-json` on both sides; mixed modes are rejected early.

Several flags are guarded to avoid ambiguous automation behavior, including partial chunks and session persistence toggles.

A stdout guard is installed before first write so accidental console output does not corrupt NDJSON framing.

Structured protocol messages are serialized through one outbound queue, preventing request/event reordering.

The `RemoteIO`/`StructuredIO` abstraction keeps stdin and stdout behavior uniform for CLI and SDK hosts.

⚠️
CI parser risk

When your downstream parser expects NDJSON, a single stray stdout line can break the run. The guard exists to prevent exactly that class of failure.

Schema mode is implemented as an internal tool, added only in non-interactive runs with `--json-schema`.

Input validation is Ajv-based, with schema-identity caching to reduce overhead across repeated scripted calls.

If the model keeps producing invalid tool payloads, the query loop exits with a typed structured-output retry error.

API format `services/api/claude.ts` sets `output_config.format` and adds structured-output beta header when supported.
Tool contract `StructuredOutput` must be called once at finalization; payload must match schema.

PLAIN ENGLISH

`--continue` means “latest local conversation”; `--resume` means “specific session identifier (UUID/JSONL/URL path).”

`--fork-session` preserves history but assigns a new session identity, preventing accidental mutation of the original transcript line.

`--resume-session-at` truncates replay up to a specific message UUID, giving deterministic branch points for automation.

1
Pick resume source

Latest (`--continue`) or explicit (`--resume`).

2
Pick identity policy

Reuse session ID or fork a new one.

3
Pick replay cursor

Optional `--resume-session-at` for partial-history continuation.

CI-safe patterns: deterministic output, explicit failure, measurable latency

`source/claude-code-source/src/cli/print.ts` (594-624, 917-973, 4838-4866), `source/claude-code-source/src/utils/headlessProfiler.ts` (62-71, 101-176), and `source/claude-code-source/src/services/api/claude.ts` (1868-1928, 2337-2364)

A
Use typed outputs

Prefer `--output-format json` for one-shot parsing or `stream-json` for event streams.

B
Bound runtime

Set `--max-turns`, `--max-budget-usd`, and optionally `--json-schema` for strict termination modes.

C
Fail loudly

Sandbox unavailability can hard-fail when required; invalid resume/format combos exit before running.

D
Instrument latency

Headless profiler checkpoints + stream watchdog detect stalls and no-event stream failures.
