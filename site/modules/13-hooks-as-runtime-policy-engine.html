Hooks as Runtime Policy Engine

Hooks are not decorative callbacks. Claude Code treats them as a policy layer that can inspect tool calls, rewrite decisions, and block continuation across session, teammate, and task boundaries.

The runtime exposes a full hook event surface

Operationally, hooks cover startup, tool calls, compaction, teammate lifecycle, file/worktree changes, and config mutation. This gives operators a central control plane for audit and intervention.

🧭

Pre-Action Gates

`PreToolUse`, `PermissionRequest`, and `Setup` can inject policy checks before side effects happen.

🧾

Post-Action Audit

`PostToolUse`, `PostToolUseFailure`, and `ConfigChange` provide durable observability after execution.

🧵

Team Lifecycle

`TaskCompleted`, `TeammateIdle`, `SubagentStart`, and `SubagentStop` enforce orchestration discipline for multi-agent runs.

🔍
Why this matters

When incidents happen, hook traces let you explain exactly why an action was blocked, allowed, or rewritten.

Pre-tool and post-tool hooks share one execution core

export async function* executePreToolHooks(...) {
  if (!hasHookForEvent('PreToolUse', appState, sessionId)) return
  const hookInput = { ...createBaseHookInput(...), hook_event_name: 'PreToolUse', ... }
  yield* executeHooks({ hookInput, toolUseID, matchQuery: toolName, ... })
}

export async function* executePostToolHooks(...) {
  const hookInput = { ...createBaseHookInput(...), hook_event_name: 'PostToolUse', ... }
  yield* executeHooks({ hookInput, toolUseID, matchQuery: toolName, ... })
}
PLAIN ENGLISH

The runtime checks hook presence first to avoid extra work when no handlers are configured.

`executeHooks(...)` is the common engine for matching, timeout, streaming progress, and blocking results.

Pre and post phases differ in payload, while execution semantics remain consistent.

Permission and config hooks are policy choke points

export async function* executePermissionRequestHooks(...) {
  const hookInput = { hook_event_name: 'PermissionRequest', tool_name, tool_input, permission_suggestions }
  yield* executeHooks({ hookInput, toolUseID, matchQuery: toolName, ... })

export async function executeConfigChangeHooks(source, filePath, timeoutMs) {
  const hookInput = { hook_event_name: 'ConfigChange', source, file_path: filePath }
  const results = await executeHooksOutsideREPL({ hookInput, matchQuery: source, timeoutMs })
  if (source === 'policy_settings') return results.map(r => ({ ...r, blocked: false }))
  return results
}
PLAIN ENGLISH

Permission hooks can pre-approve, deny, or annotate decisions before fallback prompt logic runs.

Config change hooks fire for auditing, including live watcher events during a session.

Enterprise policy settings are intentionally non-blockable, so audit and enforcement do not conflict.
