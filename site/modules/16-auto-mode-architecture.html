Auto Mode Architecture

Auto mode is classifier-mediated autonomy. Entry is gated, broad pre-approvals are stripped, cheap fast paths avoid wasted classifier calls, and failure paths fall back differently in interactive and headless runs.

Autonomy here is a guarded pipeline with four control surfaces

🚪

Entry Gate

CLI intent, settings, model support, fast-mode breaker, and org circuit breakers decide whether auto mode can even be entered.

✂️

Rule Stripping

Broad shell, PowerShell, and task allow-rules are removed before the classifier starts, so dangerous actions still reach the decision layer.

🧮

Decision Pipeline

Cheap allow paths run first; only actions with real downside reach the transcript classifier.

↩️

Exit and Recovery

When auto becomes unavailable, the runtime restores stripped rules, exits cleanly, and emits session-level notifications or attachments.

🧭

PLAIN ENGLISH

Auto mode cannot trust the old “always allow” set, because some of those rules would prevent dangerous actions from ever reaching the classifier.

The runtime actively detects broad shell/interpreter/task rules and removes them from the live permission context when auto starts.

Those removed rules are stashed so the system can restore them cleanly when auto mode ends.

⚠️
Why this matters

If `Bash(*)` or broad subagent rules stayed active, the most dangerous actions would silently bypass the very layer meant to review them.

Some actions still require explicit human review, and those are held back before any classifier call.

Cheap fast paths then allow clearly safe actions: first “would this pass in acceptEdits?”, then the safe-tool allowlist.

Only the remaining actions pay classifier latency and token cost.

Non-approvable safety checks and guarded tools stay outside auto approval.

3
Classifier path

PLAIN ENGLISH

When the classifier transcript gets too long, interactive sessions fall back toward human approval, but headless agents abort because no prompt path exists.

If the classifier is unavailable, the runtime can fail closed or fall back open depending on the gate configuration.

Repeated denials are tracked, and the system can escalate from repeated classifier blocks back into a human review flow.

Transcript too long Manual fallback in interactive mode, abort path in headless mode
Classifier unavailable Gate-controlled fail-closed deny or fail-open fallback
Repeated denials Denial tracking can force escalation back toward explicit review
