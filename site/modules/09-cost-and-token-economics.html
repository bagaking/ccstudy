Cost and Token Economics

Claude Code tracks money, throughput, and context pressure as separate ledgers. Understanding where those ledgers are computed is the difference between guessing and operating.

One session, multiple economic ledgers

In this codebase, a single turn can update several accounting systems at once: dollars, token classes, runtime duration, and code-change volume. Cost work is therefore a systems topic, not a single number on a dashboard.

💵

Billing Ledger

`totalCostUSD` and per-model USD are accumulated and formatted for terminal visibility.

🧮

Token Ledger

Input, output, cache read, cache creation, and web search requests are all tracked separately.

📏

Operational Ledger

API duration, wall duration, lines added/removed, and tool-side signals are persisted per session.

⚙️
The same response can move all three ledgers

When one assistant response includes token usage and tool actions, cost, budget pressure, and code-change analytics all move together.

PLAIN ENGLISH

Some functions count billable context with cache tokens, while others intentionally exclude cache to match specific budget formulas.

`tokenCountWithEstimation()` is the canonical threshold metric because it combines the last real usage block with estimated trailing messages.

If you choose the wrong token function, your compact or budget behavior will drift from runtime reality.

`getTokenCountFromUsage`Input + output + cache read + cache creation.
`finalContextTokensFromLastResponse`Final-window budget signal, cache excluded.
`tokenCountWithEstimation`Threshold-ready context fullness estimate across split assistant records.
