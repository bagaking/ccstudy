Context Engineering

Claude Code does not rely on one clever prompt. It assembles a runtime context stack: system prompt sections, tools, repo rules, memory files, and compaction boundaries.

Prompting is only the visible tip

The model is never handed a naked user message. By the time one turn begins, Claude Code has already layered in a system prompt, tool schemas, project instructions, memory files, and the surviving conversation history. That assembly process is what people now call context engineering.

📘

System prompt

Standing instructions, tool affordances, safety expectations, and runtime role framing.

🗂️

Repo rules

`CLAUDE.md`, `.claude/CLAUDE.md`, and rules files carry project-local instructions that shape behavior per workspace.

🧠

Memory + history

User memory, session memory, and compacted transcripts decide what the model “still knows” on the next round trip.

💡
Most “the prompt is bad” complaints are actually context-shaping bugs

If the agent ignores a repo rule or forgets a convention, the failure often sits upstream: wrong files were loaded, loaded in the wrong order, or compacted away too aggressively.

CLAUDE.md is a layered instruction system

`source/claude-code-source/src/utils/claudemd.ts` (lines 1-16)

CODE 
/**
 * Files are loaded in the following order:
 *
 * 1. Managed memory (eg. /etc/claude-code/CLAUDE.md) - Global instructions for all users
 * 2. User memory (~/.claude/CLAUDE.md) - Private global instructions for all projects
 * 3. Project memory (CLAUDE.md, .claude/CLAUDE.md, and .claude/rules/*.md in project roots) - Instructions checked into the codebase
 * 4. Local memory (CLAUDE.local.md in project roots) - Private project-specific instructions
 *
 * Files are loaded in reverse order of priority, i.e. the latest files are highest priority
 * with the model paying more attention to them.
 */
PLAIN ENGLISH

Repo instructions are not one file. They are a stack with explicit priority levels.

Managed and user memory define broader defaults; project and local files narrow behavior to the current workspace.

This is why “why did Claude behave differently here?” often reduces to “which CLAUDE.md layer won?”

1
Global policy first

Managed and user-level files establish the broadest defaults for all work.

2
Project rules next

`CLAUDE.md` and `.claude/rules/*.md` bind the agent to repo-specific conventions.

3
Nearest context wins

Files closer to the active working directory are loaded later and therefore receive more attention.

    const config = getCurrentProjectConfig()
    const includeExternal =
      forceIncludeExternal ||

    // Process Managed file first (always loaded - policy settings)

PLAIN ENGLISH

The runtime memoizes memory-file discovery because scanning rule files every turn would be wasteful.

It still preserves hierarchy: managed files, project `CLAUDE.md`, and `.claude/CLAUDE.md` all get discovered deliberately.

That means context engineering is both semantic and operational: the right files must be found, and found efficiently.
