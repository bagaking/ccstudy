    01

    From Prompt to Action

    Startup is staged on purpose. Cheap requests pay almost zero cost, and heavy systems only wake up when needed.

    The Millisecond After You Press Enter

    When you run `claude`, the process starts by reading `process.argv`. It does not load the whole agent first. It asks one routing question: is this a tiny fast-path request or a full interactive session?

    1
    Read args

    Parse command-line input and classify intent.

    2
    Branch early

    Short commands return immediately with no heavy imports.

    3
    Profile real startup

    Only non-trivial flows turn on startup checkpoints.

    Meet the Cast

    The product feels like one assistant. Under the hood, it is a baton pass across specialized actors.

    Responsibility Map, Not Blob Architecture

    Use a cast mental model: entrypoint routes, bootstrap initializes, REPL renders, QueryEngine orchestrates turns, tools execute concrete actions.

    Routing

    🚪
    cli.tsx Reads argv, classifies the request, and chooses the startup branch.

    Bootstrap

    🧰
    main.tsx Builds the runtime: permissions, tools, commands, state, and startup fan-out.

    Interaction

    🖥️
    App + REPL Owns the visible terminal shell and the live prompt/response surface.

    Execution

    🔁
    QueryEngine Runs the turn loop, dispatches tools, and feeds results back into the model.

    03

    The Conversation Loop

    One user turn can include multiple model round-trips, tool calls, and context updates before completion.

    One Turn, Many Exchanges

    The interaction model is iterative: user request, model output, tool request, tool result, model refinement, repeat. It ends on explicit stop conditions or budget limits.

    1
    Build context

    History + system prompt + user/system context + tool permission gate.

    2
    Model emits message

    Text response or structured `tool_use` request.

    3
    Execute tools

    Read-only operations can batch concurrently; mutating operations stay serial.

    4
    Append result and continue

    Tool output returns as message input for next model round.

    Concurrency Strategy for Tool Calls

    `source/claude-code-source/src/services/tools/toolOrchestration.ts` (lines 8-12, 26-36, 64-67)

    CODE
    function getMaxToolUseConcurrency(): number {
  return (parseInt(process.env.CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY || '', 10) || 10)
}

for (const { isConcurrencySafe, blocks } of partitionToolCalls(...)) {
  if (isConcurrencySafe) {
    // Run read-only batch concurrently
    for await (const update of runToolsConcurrently(blocks, ...)) { ... }
  } else {
    // Run non-read-only batch serially
    for await (const update of runToolsSerially(blocks, ...)) { ... }
  }
}
    PLAIN ENGLISH

    Concurrency defaults to 10 for safe read-only tool batches.

    Partitioning is policy-aware: concurrency-safe vs order-sensitive operations.

    Writes stay serial to preserve deterministic side-effect order.

    💡
    Tool calls are messages, not direct function privilege

    The model proposes tool calls. The runtime validates, executes, and feeds results back into the next model round.

    05

    Permissions and Trust

    High capability without explicit policy boundaries is risk. Permission design is a precision control layer, not UI ceremony.

    Permission Modes Are Trust Profiles

    defaultAsk before sensitive operations
    acceptEditsAuto-accept edit-oriented operations
    bypassPermissionsBroad auto-allow mode (high-risk)
    planPlanning mode with constrained execution behavior

    Safe configuration means tight rule scope: narrow allowlists first, broad patterns last.

    What Counts as Dangerous Bash Rule

    `source/claude-code-source/src/utils/permissions/permissionSetup.ts` (lines 94-147)

    CODE
    export function isDangerousBashPermission(toolName: string, ruleContent: string | undefined): boolean {
  if (toolName !== BASH_TOOL_NAME) return false
  if (ruleContent === undefined || ruleContent === '') return true
  const content = ruleContent.trim().toLowerCase()
  if (content === '*') return true
  for (const pattern of DANGEROUS_BASH_PATTERNS) { ... }
  return false
}
    PLAIN ENGLISH

    Blank rule content means effectively “allow everything.”

    Wildcard `*` is treated as dangerous by definition.

    Interpreter and broad execution patterns are blocked from silent auto-allow flows.

    1

    07

    Context Engineering

    Claude Code does not rely on one clever prompt. It assembles a runtime context stack: system prompt sections, tools, repo rules, memory files, and compaction boundaries.

    Prompting is only the visible tip

    The model is never handed a naked user message. By the time one turn begins, Claude Code has already layered in a system prompt, tool schemas, project instructions, memory files, and the surviving conversation history. That assembly process is what people now call context engineering.

    📘

    System prompt

    Standing instructions, tool affordances, safety expectations, and runtime role framing.

    🗂️

    Repo rules

    `CLAUDE.md`, `.claude/CLAUDE.md`, and rules files carry project-local instructions that shape behavior per workspace.

    🧠

    Memory + history

    User memory, session memory, and compacted transcripts decide what the model “still knows” on the next round trip.

    💡
    Most “the prompt is bad” complaints are actually context-shaping bugs

    If the agent ignores a repo rule or forgets a convention, the failure often sits upstream: wrong files were loaded, loaded in the wrong order, or compacted away too aggressively.

    CLAUDE.md is a layered instruction system

    `source/claude-code-source/src/utils/claudemd.ts` (lines 1-16)

    CODE 
    /**
 * Files are loaded in the following order:
 *
 * 1. Managed memory (eg. /etc/claude-code/CLAUDE.md) - Global instructions for all users
 * 2. User memory (~/.claude/CLAUDE.md) - Private global instructions for all projects
 * 3. Project memory (CLAUDE.md, .claude/CLAUDE.md, and .claude/rules/*.md in project roots) - Instructions checked into the codebase
 * 4. Local memory (CLAUDE.local.md in project roots) - Private project-specific instructions
 *
 * Files are loaded in reverse order of priority, i.e. the latest files are highest priority
 * with the model paying more attention to them.
 */
    PLAIN ENGLISH

    Repo instructions are not one file. They are a stack with explicit priority levels.

    Managed and user memory define broader defaults; project and local files narrow behavior to the current workspace.

    This is why “why did Claude behave differently here?” often reduces to “which CLAUDE.md layer won?”

    1
    Global policy first

    Managed and user-level files establish the broadest defaults for all work.

    2
    Project rules next

    `CLAUDE.md` and `.claude/rules/*.md` bind the agent to repo-specific conventions.

    3
    Nearest context wins

    Files closer to the active working directory are loaded later and therefore receive more attention.

    08

    Extensions: MCP, Skills, Plugins

    Claude Code does not grow through one extension surface. It has three: protocol-level connectors, prompt-shaped skills, and installable plugins with lifecycle management.

    Three extension surfaces, three different jobs

    🌐

    MCP

    Use this when you need the agent to talk to an external capability through a protocol: browser, database, remote system, IDE bridge.

    🧠

    Skills

    Use this when you want a reusable workflow or instruction bundle that shapes how the agent approaches a class of tasks.

    📦

    Plugins

    Use this when you want packaged distribution, install/update/uninstall flows, marketplace surfacing, and explicit lifecycle management.

    💡
    These are not interchangeable abstractions

    A workflow prompt, a remote protocol connector, and an installable package solve different problems. Most bad extension designs start by choosing the wrong surface.

    Bundled skills and plugins are registered before command loading

    `source/claude-code-source/src/main.tsx` (lines 1919-1928)

    CODE

    Extensions are part of startup topology, not a late afterthought.

    Bundled skills and built-in plugins are registered before command loading so the command layer sees the right inventory immediately.

    In other words: extension surfaces are first-class citizens in the boot sequence.
